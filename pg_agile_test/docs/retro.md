# Sprint 1 Retrospective

## 잘된 점 (What went well)
- **신속한 환경 구축:** Gemini CLI를 활용하여 Docker 데이터베이스, Python 가상환경, 프로젝트 구조 및 Agile 문서를 매우 신속하게 자동 구성했습니다.
- **TDD 프로세스 준수:** 새로운 기능(유클리드 거리 검색)을 추가할 때, 테스트를 먼저 작성하고 실패를 확인한 후 기능을 구현하여 테스트를 통과시키는 TDD 사이클을 성공적으로 따랐습니다.
- **높은 코드 품질:** 모든 테스트를 통과했으며, 코드 커버리지 100%를 달성하여 스프린트 목표를 초과 달성했습니다.

## 개선할 점 (What could be improved)
- **초기 테스트 실패:** `PYTHONPATH` 설정 누락으로 인해 모듈을 찾지 못하는 오류와, 데이터베이스 쿼리에서 파라미터를 명시적으로 타입 캐스팅하지 않아 발생한 SQL 오류가 있었습니다. 이는 초기 설정 단계에서 미리 예측하고 방지할 수 있었습니다.
- **테스트 격리성:** 현재 테스트는 데이터베이스 상태를 공유하고 있어 실행 순서에 따라 결과가 달라질 수 있는 잠재적 위험이 있습니다. 각 테스트가 독립적으로 실행될 수 있도록 개선이 필요합니다.

## 다음 스프린트를 위한 액션 아이템 (Action items for next sprint)
- **테스트 실행 환경 개선:** `pytest` 실행 시 `PYTHONPATH`가 자동으로 설정되도록 `pytest.ini` 또는 유사한 설정 파일을 구성하는 것을 고려합니다.
- **`pytest` Fixture 도입:** 각 테스트 함수가 독립적인 데이터베이스 상태를 가질 수 있도록, 테스트 실행 전 데이터를 준비하고 실행 후 정리하는 `pytest` Fixture를 도입합니다.
- **API 계층 추가:** 현재는 테스트 코드가 `db_utils`를 직접 호출하지만, 다음 스프린트에서는 Flask를 사용하여 간단한 API 엔드포인트를 만들고, 해당 엔드포인트를 테스트하는 상위 레벨의 테스트를 추가합니다.
