# 스프린트 회고 (Sprint 1 Retrospective)

- **날짜:** 2025년 11월 19일
- **참석자:** Gemini CLI

---

## 1. 잘된 점 (What went well)

- **신속한 초기 세팅:** 주어진 요구사항 명세에 따라 `docker-compose.yml`, `requirements.txt` 등 프로젝트의 기본 구조와 파일들을 매우 빠르게 생성할 수 있었다.
- **명확한 목표 달성:** 스프린트 목표였던 "기본 환경 구축 및 핵심 시나리오 실행 자동화"를 완수하고, 실제 테스트 실행까지 성공적으로 마쳤다.
- **자동화된 테스트 실행:** `test_runner.py`가 `scenarios` 폴더의 모든 테스트를 동적으로 발견하고 실행하여, Redis와 Valkey의 동작을 성공적으로 비교하고 결과를 출력했다.
- **일관된 결과 확인:** 테스트 결과, `pubsub` 시나리오를 제외한 모든 기능이 두 데이터베이스에서 동일하게 동작함을 확인하여 호환성을 검증하려는 초기 목표를 달성했다.

## 2. 개선할 점 (What could be improved)

- **`pubsub` 테스트 실패:** `pubsub` 시나리오가 Redis와 Valkey 양쪽 모두에서 "Subscriber did not receive a message" 오류로 실패했다. 이는 테스트 로직 자체의 문제일 가능성이 높다. (예: `time.sleep` 값이 너무 짧아 메시지 수신 전에 테스트가 종료됨). 테스트 코드의 안정성을 더 높여야 한다.
- **실행 환경 설정:** Python 가상 환경 설정 및 의존성 설치 과정을 사용자가 직접 개입하여 바로잡아야 했다. `README.md`에 명시된 절차를 처음부터 정확하게 따라 실행했다면 더 원활했을 것이다.
- **컨텍스트 파악 오류:** 초기 `docs` 디렉토리 생성 시도나, 애자일 문서의 컨텍스트가 다른 점 등, 주어진 환경과 요구사항을 더 면밀히 분석하고 진행하는 세심함이 필요하다.

## 3. 다음 스프린트를 위한 액션 아이템 (Action Items for Next Sprint)

- **(실행)** `pubsub.py` 테스트 시나리오를 디버깅한다. `time.sleep` 값을 늘리거나, `pubsub.get_message()`의 동작 방식을 재검토하여 안정적인 테스트로 수정한다.
- **(실행)** 새로운 프로젝트 수행 시, `README.md` 또는 설정 가이드에 명시된 절차를 그대로 따라서 환경을 구성하고, 임의로 단계를 건너뛰지 않는다.
- **(고려)** 테스트 실패 시, 실패 원인이 "두 대상의 동작 차이" 때문인지, "테스트 코드 자체의 결함" 때문인지 명확히 구분하여 리포트하는 로직을 `test_runner.py`에 추가하는 것을 고려한다.
